# A Framework For System Design Interviews

시스템 설계 인터뷰를 효과적으로 진행하기 위한 4단계 프레임워크.

## 인터뷰어가 평가하는 것

- 기술적 설계 능력
- **협업** 능력
- 압박 하에서 **문제 해결** 능력
- 모호함을 건설적으로 **해소**하는 능력
- 좋은 **질문**을 하는 능력

### Red Flags
- Over-engineering (과잉 설계)
- 편협한 사고
- 고집

## 4단계 프로세스

### Step 1: 문제 이해 및 설계 범위 설정 (3-10분)

**절대 바로 답을 내지 마라.** 요구사항을 철저히 이해하지 않고 답을 내는 것은 큰 실수.

질문 예시:
- 어떤 기능을 구축하는가?
- 사용자 수는?
- 예상 확장 규모는? (3개월, 6개월, 1년)
- 기술 스택은? 활용 가능한 기존 서비스는?

```
예시 (뉴스 피드 시스템):
Q: 모바일 앱? 웹? 둘 다?        → 둘 다
Q: 가장 중요한 기능?              → 포스팅 + 피드 보기
Q: 피드 정렬 순서?               → 역시간순
Q: 친구 수 제한?                 → 5,000
Q: 트래픽 볼륨?                  → DAU 1,000만
Q: 미디어 포함?                  → 이미지 + 동영상
```

### Step 2: 고수준 설계 제안 및 동의 얻기 (10-15분)

- 초기 설계 청사진 작성 후 **인터뷰어 피드백** 요청
- 핵심 컴포넌트를 박스 다이어그램으로 그리기
  - 클라이언트, API, 웹 서버, 데이터 저장소, 캐시, CDN, 메시지 큐
- Back-of-the-envelope 계산으로 규모 제약 확인
- 구체적인 use case를 통해 고수준 설계 검증

### Step 3: 상세 설계 (10-25분)

- 인터뷰어와 함께 **우선순위 컴포넌트** 식별
- 인터뷰어의 힌트에 따라 집중 영역 결정
- **시간 관리** 필수 - 불필요한 세부사항에 빠지지 말 것

### Step 4: 마무리 (3-5분)

논의할 수 있는 주제:
- 시스템 **병목 현상** 및 개선 방안
- 설계 **요약**
- **오류 케이스** (서버 장애, 네트워크 손실)
- **운영 이슈** (모니터링, 로깅, 배포)
- **다음 확장 단계** (100만 → 1000만 사용자)

## Do's and Don'ts

### Do's
- 항상 **명확화 질문**을 하라
- 요구사항을 **이해**하라
- **여러 접근법**을 제안하라
- 가장 **중요한 컴포넌트**부터 설계하라
- 인터뷰어와 **아이디어를 나눠라**
- 절대 **포기하지 마라**

### Don'ts
- 준비 없이 가지 마라
- 요구사항 확인 없이 솔루션에 뛰어들지 마라
- 처음부터 하나의 컴포넌트에 너무 깊이 들어가지 마라
- 막히면 **힌트를 요청**하라
- 침묵 속에서 생각하지 마라
- 인터뷰어가 끝이라고 할 때까지 끝이 아니다

## 시간 배분 (45분 기준)

| 단계 | 시간 |
|------|------|
| Step 1: 문제 이해 | 3-10분 |
| Step 2: 고수준 설계 | 10-15분 |
| Step 3: 상세 설계 | 10-25분 |
| Step 4: 마무리 | 3-5분 |
